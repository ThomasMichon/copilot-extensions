# Agent-Bridge Adopt

Wire agent-bridge to a repository's machine topology. This creates a
**topology profile** in `~/.agent-bridge/config.yaml` that tells
agent-bridge where to find `machines.yaml` and `acp-agents.json` for
the project.

**Prerequisite:** agent-bridge must be installed first (see the
`agent-bridge-init` skill).

## What It Does

Adds or updates a topology profile in `~/.agent-bridge/config.yaml`:

```yaml
topologies:
  facility:                          # profile name
    machines_yaml: /path/to/repo/machines.yaml
    agents_config: /path/to/repo/tools/mcp/acp-agents.json
```

This wires agent-bridge to the **same machines** that appear in
Windows Terminal fragments and SSH config. The source of truth is
`machines.yaml` -- both agent-bridge and agent-worktrees derive
their machine awareness from this single file.

## How to Run

### Option A: CLI command (preferred)

```bash
# Auto-discovers machines.yaml and acp-agents.json in the repo
agent-bridge config adopt --repo /path/to/repo --profile facility

# Verify
agent-bridge config show
agent-bridge config validate
```

### Option B: Manual (for custom layouts)

```bash
agent-bridge config adopt \
  --repo /path/to/repo \
  --profile facility \
  --machines-yaml /path/to/machines.yaml \
  --agents-config /path/to/acp-agents.json
```

## Auto-Discovery Paths

The `config adopt` command searches these conventional locations:

**machines.yaml:**
```
{repo}/machines.yaml
{repo}/config/machines.yaml
{repo}/.github/machines.yaml
```

**acp-agents.json:**
```
{repo}/tools/mcp/acp-agents.json
{repo}/acp-agents.json
{repo}/config/acp-agents.json
```

## Adoption Flow

### 1. Detect repo identity

From the current working directory:

```bash
repo_root=$(git rev-parse --show-toplevel)
```

### 2. Choose profile name

The profile name is how topology is referenced in API calls. Use a
descriptive name matching the project:

- `facility` -- for single-project machines
- `aperture-labs` -- project-specific
- `dotfiles` -- project-specific

If only one topology exists, agent-bridge uses it as the default.

### 3. Run adopt

```bash
agent-bridge config adopt --repo "$repo_root" --profile facility
```

### 4. Restart the service

Topology changes require a service restart:

```bash
# Direct restart
agent-bridge-install stop && agent-bridge-install start

# Or via systemd (Linux)
systemctl --user restart agent-bridge.service

# Or via the install script
install.ps1 stop; install.ps1 start   # Windows
install.sh stop; install.sh start     # Linux
```

### 5. Verify

```bash
# Check topology loaded
agent-bridge machines
agent-bridge agents

# Validate config
agent-bridge config validate
```

## Managing Multiple Repos

Each repo gets its own topology profile:

```bash
agent-bridge config adopt --repo ~/src/aperture-labs --profile aperture-labs
agent-bridge config adopt --repo ~/src/dotfiles --profile dotfiles
```

Result in `config.yaml`:

```yaml
topologies:
  aperture-labs:
    machines_yaml: /home/user/src/aperture-labs/machines.yaml
    agents_config: /home/user/src/aperture-labs/tools/mcp/acp-agents.json
  dotfiles:
    machines_yaml: /home/user/src/dotfiles/machines.yaml
    agents_config: /home/user/src/dotfiles/agents.json
```

## Removing a Profile

```bash
agent-bridge config remove my-profile
```

## Relationship to Terminal Fragments

Windows Terminal fragments (generated by agent-worktrees) and
agent-bridge topology both derive from `machines.yaml`. They are
two views of the same machine registry:

- **Terminal fragments** provide SSH profiles for interactive sessions
- **Agent-bridge topology** provides programmatic agent dispatch

When adopting a repo for agent-bridge, the config will point to the
same `machines.yaml` that agent-worktrees uses for terminal generation.
If the file path changes (e.g., repo moves), both systems need updating.

## Troubleshooting

- **"No machines.yaml or acp-agents.json found"** -- the repo doesn't
  have topology files at conventional locations. Specify paths explicitly
  with `--machines-yaml` and `--agents-config`.
- **"agent-bridge has no agents"** -- topology profile exists but the
  service needs a restart to load it. Restart the service.
- **Agents list empty after restart** -- check `agent-bridge config validate`
  to verify the config file paths are correct and the files exist.
- **Machines don't match terminal profiles** -- both should point to the
  same `machines.yaml`. Check the topology profile path vs the
  agent-worktrees project config.
